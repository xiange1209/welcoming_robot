## 一、專案目標

以 ROS 2 為唯一主架構，使用 Raspberry Pi 4 與筆電分工協作，完成以下專題目標：

1. 以單張大頭照完成 VIP / 黑名單註冊（含 gender、person_type 欄位），並在現場辨識 VIP、一般訪客與黑名單。
2. 語音輸入經喚醒詞 + ASR 轉為文字（`/user_text`）後送入 LLM Agent 分析，由 Agent 呼叫 ROS2 工具（地圖/地點/導航）控制機器人。
3. 以 ROS 2 串接 vision、audio、llm、navigation 等子系統，由 smartnav_bringup 一鍵啟動。
4. 針對銀行場景增強 LLM，讓它能處理迎賓決策、安全通報與金融問答（目前工具集尚未涵蓋，見已知限制）。
5. 在機器人端提供人機介面，讓使用者或組員能看到辨識結果與系統狀態（smartnav_hmi 目前為空殼預留）。

### 雙機架構

- **RPi4**（Ubuntu 24.04、ROS2 Jazzy、8GB RAM）：跑 vision / audio / navigation / bringup + TurtleBot3。
- **Windows 筆電**（RTX 3050 4GB）：跑 Ollama；RPi4 的 smartnav_llm 透過 `ollama_base_url` 參數連過來。

```bash
ros2 launch smartnav_bringup smartnav.launch.py ollama_base_url:=http://192.168.1.xxx:11434
```

smartnav.launch.py 的 launch 參數：`image_topic` / `enable_gpu` / `ollama_base_url` / `model_name` / `use_sim_time` / `enable_vision` / `enable_audio` / `enable_llm` / `enable_navigation` / `enable_nav2`（預設 false）。

### LLM 決策架構（2026-07-02 更新：唯一路線）

筆電 FastAPI 路線（原 `src/llm_server/`）已於 2026-07-02 正式廢止並刪除。**LLM 決策唯一路線是 smartnav_llm（LangChain Agent + Ollama 工具呼叫）**：

- `llm_service_node` 訂閱 `/user_text`（std_msgs/String）。
- 參數：`ollama_base_url`（預設 `http://localhost:11434`）、`model_name`（預設 `gemma4:e2b`）、`temperature`（0.0）。
- 工具集（**全為導航導向**）：create_map / list_maps / switch_map / create_waypoint / list_waypoints / navigate / global_localization。
- 現階段運算安排：RPi4 沒有 GPU，Ollama 跑在筆電（RTX 3050）上，透過 `ollama_base_url:=http://<筆電IP>:11434` 指定；未來若有本機推理能力，改回 `http://localhost:11434` 即可，不需改邏輯。

**已知限制（誠實揭露）**：smartnav_llm 的工具集全為導航導向，**沒有銀行場景工具**（例如「帶 VIP 到貴賓室」「通報人員處理黑名單」）；`config/system_prompt.txt` 也是導航機器人設定。人臉辨識事件經 recognition_text_bridge 進入 `/user_text` 後，目前只會被當成一般對話處理，不會觸發銀行專屬決策——這是後續必須擴充的部分。

**模型建議（依 2026-07-02 實測更新）**：雖然 `model_name` 預設為 `gemma4:e2b`，但在 RTX 3050 4GB 筆電上實測，gemma4 系列模型檔（7.2GB / 9.6GB）超出 4GB VRAM，被 Ollama 卸載到 CPU，延遲慢 20~40 倍，不適合即時互動。此硬體上建議改用 `model_name:=qwen2.5:3b`（實測最快）或 `gemma3:4b`（次之），實測數據見「七、驗證矩陣」。

### 目前尚未整合 / 尚未驗證的部分

- **整個 ROS2 workspace 尚未在 RPi4 實機執行過 colcon build / 運行**——vision→LLM 橋接、bringup、audio、navigation 皆屬「已實作、待實機驗證」。
- smartnav_llm 缺少銀行場景工具與 system prompt（見上方已知限制）。
- `frontier_exploration_ros2` 需另外 clone（https://github.com/mertgulerx/frontier_exploration_ros2 ）到 `src/`；同學 repo 是用 git submodule 引入，本 repo 尚未加。
- 活體檢測（liveness）：舊原型的 liveness_detector.py 已隨重組刪除，ROS2 版目前沒有對應功能，屬未來擴充項。

## 七、驗證矩陣

### 已驗證

| 項目 | 結論 |
|------|------|
| RPi4 即時人臉檢測 | InsightFace buffalo_sc 達 15+ FPS（於舊架構驗證；辨識引擎與 ROS2 版 face_engine 相同，故結論可沿用，但 ROS2 節點本身仍需實機驗證） |
| LLM 模型延遲基準（2026-07-02，筆電實測） | 見下表 |

**LLM 延遲實測**（Windows 筆電 RTX 3050 Laptop 4GB VRAM、Ollama 0.24.0、`scripts/benchmark_llm_models.py`、每模型 1 次暖機 + 5 次正式、串流 /api/chat、銀行場景 system prompt、temperature 0.1、num_ctx 512；原始明細在 `benchmark_llm_results.json`）：

| 模型（Ollama tag） | 基底 | 首次輸出平均 (ms) | 總生成平均 (ms) | 最快 (ms) | 最慢 (ms) | 成功率 |
|---|---|---:|---:|---:|---:|---:|
| gemma3-bank | gemma3:4b | 2303 | 4700 | 4583 | 4755 | 100% |
| gemma4e2b-bank | gemma4:e2b | 42235 | 47909 | 44838 | 50206 | 100% |
| gemma4e4b-bank | gemma4:e4b | 95066 | 107123 | 100231 | 123839 | 100% |
| qwen2.5:3b | qwen2.5:3b | 2218 | 3076 | 3017 | 3196 | 100% |

關鍵解讀：gemma4 模型檔 7.2GB / 9.6GB 超出 4GB VRAM，被 Ollama 卸載到 CPU 導致慢 20~40 倍；gemma3:4b（4GB）與 qwen2.5:3b（1.9GB）能塞進 VRAM。在此硬體上 **qwen2.5:3b 最快、gemma3:4b 次之，gemma4 系列不適合即時互動**。

## 八、近期建議任務

### 最值得先做的 3 件事（2026-07-02 更新）

1. **在 RPi4 上 colcon build 整個 workspace**：ROS2 Jazzy 環境執行 `colcon build --symlink-install`，確認 8 個套件（特別是 smartnav_msgs 介面產生與 vision/audio/llm 的 Python 相依）都能成功建置，順手整理出可重現的相依安裝步驟。
2. **人臉註冊 → 辨識 → `/user_text` 橋接實機驗證**：用 `ros2 service call /face_registration/register` 註冊 VIP/黑名單，確認 face_recognition_node 發布 `/face_recognition/result`（`ros2 topic echo` 檢查 gender/person_type/confidence），再確認 recognition_text_bridge 有把事件轉成自然語言發到 `/user_text`。
3. **smartnav_llm 連筆電 Ollama 實測**：`ros2 launch smartnav_llm llm.launch.py ollama_base_url:=http://<筆電IP>:11434 model_name:=qwen2.5:3b`，驗證 Agent 能連線並正確呼叫導航工具。**建議先用 qwen2.5:3b**——2026-07-02 實測它在 RTX 3050 4GB 上最快（首次輸出 2.2 秒、總生成 3.1 秒），預設的 gemma4:e2b 在此硬體會慢 20~40 倍。

### 可以之後再做的項目

2. 為 smartnav_llm 新增銀行場景工具（帶位、通報）並改寫 system_prompt.txt。
3. 把 frontier_exploration_ros2 以 submodule 或直接 clone 方式加進 src/，補齊自動探索建圖。
4. 在硬體到位前，先用模擬環境（Gazebo，nav2.launch.py 已引用 turtlebot3_gazebo）驗證地圖與導航流程。
5. 接入銀行知識與匯率資料來源。

## 九、基於目前設備的建議實作順序

考量目前只有 RPi4、鏡頭與 RTX 3050 4GB 筆電，最務實的順序如下：

1. 先在 RPi4 完成 colcon build，單獨啟動 smartnav_vision 驗證人臉註冊與辨識（`/face_recognition/result`）。
2. 接上 recognition_text_bridge 與 smartnav_llm（筆電 Ollama，qwen2.5:3b），驗證「看到 VIP → `/user_text` → LLM 回應」這條最短鏈。
3. 下載語音模型，驗證 smartnav_audio 的喚醒 + ASR + TTS。
4. 等底盤 / 導航感測器或模擬環境到位後，再實測地圖建立與導航（smartnav_navigation + frontier_exploration_ros2）。
5. 最後再做銀行場景工具、smartnav_brain 編排與 smartnav_hmi 介面。

## 十、風險與限制

- 目前最關鍵的風險不是演算法，而是**整個 ROS2 workspace 尚未在 RPi4 實機 colcon build / 運行過**；重組刪除舊原型後，已沒有可退回的單機展示路線，第一次實機建置務必盡早進行。
- smartnav_llm 預設模型 gemma4:e2b 在現有筆電（RTX 3050 4GB）上實測不可用於即時互動（首次輸出 42 秒起跳），啟動時必須明確指定 `model_name:=qwen2.5:3b` 或 `gemma3:4b`。
- 人臉事件進 `/user_text` 後只會被當一般對話：工具集無銀行場景工具、system prompt 為導航設定，「辨識 → 迎賓/通報動作」這條語意層目前是最大的功能斷點。
- frontier_exploration_ros2 尚未加入本 repo（同學用 git submodule），忘記 clone 會導致建圖相關 launch 失敗。
- smartnav_audio、smartnav_llm、smartnav_navigation 依賴大量外部套件（sherpa-onnx、langchain/langchain-ollama、Nav2 全家桶、slam_toolbox），舊的 requirements/setup 腳本已刪除，需在第一次 colcon build 時同步整理出新的相依安裝流程。
- 單樣本辨識目前是可運作概念，不等於已證明在實際銀行光線、姿態下有足夠準確率。
- 地圖建立、跟隨、帶領等功能受限於硬體，若沒有移動底盤與導航感測器，只能先做模擬或介面設計。

## 十一、建議的進度更新格式

之後每次更新進度時，建議固定用這四欄，最不容易失焦：

1. 已完成：程式存在且已驗證。
2. 待驗證：程式存在但沒有實機證據。
3. 進行中：正在做且有明確下一步。
4. 阻塞：受限於硬體、環境或外部服務。

## 十二、現階段建議展示路線

若目標是近期要向組員或老師展示，建議走這條最短成功路徑：

1. 筆電啟動 Ollama 並拉取 `qwen2.5:3b`（實測最快的模型）。
2. RPi4 完成 colcon build 後，啟動總入口（可先關閉尚未驗證的模組，只跑 vision + llm）：
   ```bash
   ros2 launch smartnav_bringup smartnav.launch.py \
     ollama_base_url:=http://<筆電IP>:11434 model_name:=qwen2.5:3b \
     enable_audio:=false enable_navigation:=false
   ```
3. 用 `/face_registration/register` 服務註冊 VIP 與黑名單，展示辨識結果（`/face_recognition/result`、debug image 的綠/紅/藍框）。
4. 展示 recognition_text_bridge → `/user_text` → LLM 回應的橋接對話。

這條路徑完成後，再展示語音鏈（smartnav_audio）與導航/建圖（smartnav_navigation），並誠實說明這些目前是「程式碼已完成、待實機驗證」，且 LLM 的銀行場景工具（帶位、通報）尚未實作。
