## 一、專案目標

以 ROS 2 為最終主架構，使用 Raspberry Pi 4 與筆電分工協作，完成以下專題目標：

1. 以單張大頭照完成 VIP / 黑名單註冊，並在現場辨識 VIP、一般訪客與黑名單。
2. 將多語言語音輸入送入 LLM 分析，輸出固定 JSON 給 ROS 2 節點做控制。
3. 以 ROS 2 串接 vision、brain、audio、robot action、front-end UI 等子系統。
4. 針對銀行場景增強 LLM，讓它能處理金融問答、匯率等資訊查詢與迎賓決策。
5. 在機器人端提供 UI，讓使用者或組員能看到辨識結果、系統狀態與功能頁面。

### smartnav_vision 版本確認（已決策，2026-07-01 第二輪更新）

之前擔心的「兩版本分岔」風險已解除。當初只用 Compare-Object 比對差異行數，未確認內容；現已直接從孫同學維護的 GitHub 專案（https://github.com/swient/smartnav-bot）逐函式讀取完整原始碼並比對，確認：

- 本專案目前的 `face_engine.py`、`database_manager.py`、`face_recognition_node.py`、`face_registration_node.py` 都是在他的測試版基礎上，正確地加上 gender、person_type（VIP/BLACKLIST）、黑名單優先辨識邏輯，不是未確認的分岔版。
- `face_utils.py` 與他的版本完全相同，無需變動。
- 唯一真正缺失的是 `setup.py` 缺少 `install_requires`、`package.xml` 少兩行 `exec_depend`：已在 2026-07-01 修正，補齊 `insightface`、`opencv-python`、`numpy`、`onnxruntime` 相依宣告。

結論：**本專案 smartnav_vision 保持不覆蓋**，因為它已經是「孫同學測試過的基礎 + 我們的銀行客製化」正確疊加後的結果，不是需要重寫或提升風險的分岔。

### LLM 決策架構（已決策，2026-07-01 第二輪更新）

終局架構確定為 **ROS2 smartnav_llm（LangChain Agent + 工具呼叫）**，直接控制地圖/地點/導航等 ROS2 服務。目前仍有兩條並行路徑，但角色已明確：

1. **筆電 FastAPI + Ollama**（`src/llm_server/`）：固定 JSON schema（intent/action/target/reply），保留作為非 ROS2 單程序驗證流程（Phase 2b/3a）的並行橋接，**不是終局架構**，但目前仍有實用價值，不會刪除。
2. **ROS2 smartnav_llm**（`src/smartnav_ros2/smartnav_llm/`）：**終局架構**。LangChain Agent 直接呼叫 ROS2 的地圖/地點/導航服務，模型預設 gemma4:e2b。

**現階段運算安排**：RPi4 沒有 GPU，所以 smartnav_llm 實际呼叫的 Ollama 仍需要跑在筆電（RTX 3050）上。`llm_service_node` 新增了 `launch/llm.launch.py`，可用 `ollama_base_url` 參數指定筆電的區網 IP：

```bash
ros2 launch smartnav_llm llm.launch.py ollama_base_url:=http://<筆電IP>:11434
```

待 RPi4 或其他邏輯需要本機推理能力時，只需把此參數改回 `http://localhost:11434` 即可，不需改 smartnav_llm 的邏輯。

### 目前尚未整合進主流程的部分

- src/ai_vision/liveness_detector.py：程式存在，但尚未完成 ARM64 驗證
- smartnav_vision（ROS2 版）尚未與 smartnav_llm / smartnav_brain 串接（人臉辨識結果目前不會觸發導航或語音）
- smartnav_audio、smartnav_brain、smartnav_llm 皆尚未在 RPi4 / TurtleBot3 實機測試過，僅完成程式碼整併
- Whisper STT / TTS（RPi4 單程序版）：已改用 ROS2 版的 sherpa-onnx 方案，舊規劃可視為取消

## 三、功能/架構對照表（預期 vs 目前狀態 vs 差異）

這是本文件的核心表格，逐項對照使用者原始需求、目前 repo 實際狀態，以及還需要加強或新增的部分。

| 功能/架構 | 預期需求 | 目前狀態 | 差異／待加強 |
|---|---|---|---|
| 人臉辨識－分類 | 單張大頭照即可辨識 VIP／一般訪客／黑名單，並觸發對應動作 | 已確認以孫同學測試過的 GitHub 版本為基礎，正確疊加本專案的 gender、person_type（VIP/BLACKLIST）分類（非分岔） | 辨識後的「觸發對應動作」尚未接到 ROS2 導航/語音 |

| 銀行知識 / 金融資訊強化 | 銀行問答、匯率等金融資訊查詢 | 僅有 system prompt 層級規則，無 RAG、無外部資料源 | 尚未開始：需接入匯率 API 或銀行知識庫，由 ROS2 smartnav_llm 負責（依上方決策） |
| ROS2 機器人控制與動作 | 帶領、跟隨、語音提示等行為 | smartnav_brain 已有 Navigate action、地點（waypoint）管理、地圖服務 | 只有「導航到指定地點」，尚無「帶領」「跟隨」語意層；尚未與人臉辨識/LLM 串接觸發 |
| 地圖建立 | 可用語音下指令建立/控制地圖 | smartnav_brain 已有 CreateMap action，並整合 frontier_exploration_ros2 自動探索建圖 | 尚未與語音喚醒/LLM Agent 串接觸發；尚未在實體機器人上測試（目前設備缺移動底盤與導航感測器） |

| 系統整合（bringup） | 各模組可一鍵啟動並互相串接 | smartnav_bringup 仍是空殼；smartnav_brain 有 brain.launch.py / nav2.launch.py，但只涵蓋地圖/導航，未涵蓋 vision/audio/llm | 尚無涵蓋全部 5 個 ROS2 套件的總 launch；vision/audio/llm 尚未接上 waypoint/navigate 等服務 |

| smartnav_llm 接上 smartnav_vision | 目前 smartnav_vision 的辨識結果不會觸發 smartnav_brain 導航或 smartnav_llm 對話，需要新增橋接邏輯（例如 vision 發布辨識結果 topic，llm 訂閱後轉成 user_text 或直接觸發工具） |
| ROS2 bringup 總入口 | 尚無同時啟動 vision + audio + llm + brain 的總 launch |
| TurtleBot3 自主導航 / 地圖建立實機測試 | 硬體目前缺移動底盤，只能先做程式碼驗證或模擬環境 |

| 金融資訊查詢 / 匯率整合 | 尚未接入 smartnav_llm 決策流程 |
| 機器人本體 UI | 尚未接到 RPi4 或 ROS2 主流程 |

## 八、近期建議任務

### 最值得先補的 3 件事（2026-07-01 第二輪更新）

1. 在真正的 ROS2 Jazzy 環境跑一次 `colcon build --symlink-install`，確認整併後的 smartnav_audio / smartnav_brain / smartnav_llm / smartnav_msgs / smartnav_vision 五個套件都能成功建置（含 rosdep 相依）。
2. 啟動 smartnav_llm（`ros2 launch smartnav_llm llm.launch.py ollama_base_url:=http://<筆電IP>:11434`），驗證 LangChain Agent 能否連上筆電的 Ollama 並正確呼叫地圖/地點/導航工具。
3. 把 smartnav_vision 的辨識結果接進 smartnav_llm 或 smartnav_brain，讓 VIP/黑名單/訪客能真正觸發導航或語音行為（目前最大的斷點）。

### 可以之後再做的項目

1. 下載 sherpa-onnx 語音模型（download_audio_models.sh 邏輯），在 RPi4 驗證喚醒詞、ASR、TTS 延遲。
2. 接入銀行知識與匯率資料來源，接進 smartnav_llm 的工具鏈。
3. 補一個涵蓋 vision + audio + llm + brain 的 smartnav_bringup 總 launch。
4. 在有足夠硬體前，先用模擬環境（Gazebo，nav2.launch.py 已引用 turtlebot3_gazebo）驗證地圖與導航流程。

## 九、基於目前設備的建議實作順序

考量目前只有 RPi4、鏡頭與 RTX 3050 4GB 筆電，最務實的順序如下：

1. 先把人臉辨識 + LLM JSON + 基本 Web UI 這條非 ROS2 主線做穩（Phase 2b / 3a 驗證）。
2. 在 ROS2 環境驗證 smartnav_audio / smartnav_brain / smartnav_llm 是否能各自單獨建置與啟動。
3. 把 smartnav_vision 的辨識結果接進 smartnav_llm（終局 LLM 決策架構），完成人臉辨識到 ROS2 決策的橋接。
4. 等底盤 / 導航感測器或模擬環境到位後，再實測地圖建立、帶領與跟隨。
5. 最後才把完整 UI 放到機器人端，整合所有模組狀態。

## 十、風險與限制

- 目前最關鍵的風險不是演算法，而是缺少完整實機驗證紀錄；這次整併大幅增加了「代碼存在」的範圍，但沒有增加「已驗證」的範圍。
- benchmark.py 雖已對齊現行模組路徑，但若沒有 ONNX 模型檔，無法得到完整模型基準。
- Gemma 系列的效能數據目前仍屬規劃或估計值，不應當成既有成果展示。
- smartnav_llm（終局 LLM 架構）與 smartnav_vision（人臉辨識）尚未串接，這是目前通往「完整展示」最大的單一斷點。
- smartnav_audio、smartnav_brain、smartnav_llm 依賴大量外部套件（sherpa-onnx、langchain/langchain-ollama、nav2 全家桶、slam_toolbox、frontier_exploration_ros2），這些都不在目前 requirements_minimal.txt / setup_ubuntu.sh 涵蓋範圍內，需要另外規劃 ROS2 端的安裝流程。
- 單樣本辨識目前是可運作概念，不等於已證明在實際銀行光線、姿態下有足夠準確率。
- 地圖建立、跟隨、帶領等功能目前受限於硬體，若沒有移動底盤與導航感測器，只能先做模擬或介面設計。

## 十一、建議的進度更新格式

之後每次更新進度時，建議固定用這四欄，最不容易失焦：

1. 已完成：程式存在且已驗證。
2. 待驗證：程式存在但沒有實機證據。
3. 進行中：正在做且有明確下一步。
4. 阻塞：受限於硬體、環境或外部服務。

## 十二、現階段建議展示路線

若目標是近期要向組員或老師展示，建議只展示這條最短成功路徑：

1. 筆電啟動 qwen2.5:3b + FastAPI Server。
2. RPi4 啟動人臉即時辨識。
3. 先證明 VIP / 黑名單 / 訪客標籤正常。
4. 再補上筆電端 /chat 或 /api/chat 回應展示。

這條路徑完成後，再考慮展示 ROS2 端目前已整併但尚未串接的能力：語音喚醒＋辨識、地圖建立、LangChain Agent 控制導航（`ros2 launch smartnav_llm llm.launch.py ollama_base_url:=http://<筆電IP>:11434`），並說明這些目前是「程式碼已完成、待整合與實機驗證」而非「已可展示的完整流程」。
