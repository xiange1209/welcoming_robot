# 實驗設計：紙本報告與競賽用的量化數據

建立日期：2026-07-17。目的：專題報告不能只有「功能會動」，要有**可比較、可重現的數據**。本文件定義八組實驗（E1~E8），每組含目的、方法、變因、樣本數、記錄表模板。做完一組就把數據填回本檔或另存 CSV，並在狀態欄改「已完成＋日期」。

**共同原則**：每個數字記錄當下的軟硬體條件（模型版本、參數、環境）；失敗的嘗試也記（誠實原則）；照片／截圖存 `實驗記錄/E<編號>/`（目錄不進 git，最終數據表進報告）。

## E1. 人臉辨識準確率矩陣 ⭐報告主菜

- **狀態**：未開始（Phase B；有 USB webcam 可在家先做）
- **目的**：單樣本註冊在不同條件下的辨識率——回答企劃書原始需求「單張照片穩定辨識」到底行不行
- **方法**：註冊 VIP 1 人、黑名單 1 人（各 1 張正面照）；受試者站定點，記錄 `/face_recognition/result` 連續 20 幀的辨識結果與 confidence
- **變因**：距離 1m／2m／3m × 角度 正面／30°／45° × 光線 日光燈全開／半暗
- **判定**：辨識率 = 正確幀數/20；同時記錄「誤認為他人」與「未偵測」分開統計
- **產出**：18 格條件矩陣表＋confidence 箱型圖＋**三類混淆矩陣**（VIP／訪客／黑名單互相誤認率——安全系統的關鍵指標是「黑名單漏報率」與「訪客誤報為黑名單率」）

記錄表模板（每格條件一列）：

| 距離 | 角度 | 光線 | 正確幀 | 誤認幀 | 未偵測幀 | confidence 平均±標準差 |
|---|---|---|---|---|---|---|
| 1m | 正面 | 亮 | /20 | | | |

## E2. 註冊樣本數比較

- **狀態**：未開始（Phase B，與 E1 同場收）
- **目的**：`num_samples` 1／5／10 對辨識率的影響——論證「單樣本」的取捨
- **方法**：同一受試者分別以 1／5／10 樣本註冊（每次先清資料庫），跑 E1 的「2m×正面×亮」條件 20 幀
- **產出**：樣本數 vs 辨識率／confidence 折線；報告可寫「單樣本已達 X%，多樣本增益 Y%」

## E3. LLM 工具呼叫成功率

- **狀態**：未開始（Phase B，銀行工具做完後）
- **目的**：補 LLM 規劃書「工具呼叫成功率：待測」欄；比較兩個可用模型
- **方法**：`qwen2.5:3b` 與 `gemma3:4b` × 3 情境 × 20 次。情境：(a)「帶我去貴賓室」→應呼叫 guide_to_vip_room；(b)「現在幾點開門」→應呼叫 query_bank_faq 而非亂答；(c) 閒聊「你好」→應**不**呼叫工具直接回覆
- **判定**：正確呼叫（或正確不呼叫）＝成功；記錄多步 ReAct 的步數
- **產出**：2×3 成功率表＋失敗案例分類（幻覺工具名／參數錯誤／該呼叫沒呼叫）

## E4. LLM 延遲 ✅（已有）＋端到端任務延遲（延伸）

- **狀態**：四模型延遲**已完成**（2026-07-02，`benchmark_llm_results.json`、`scripts/benchmark_llm_models.py`，報告直接引用）；端到端延伸未開始（Phase C）
- **既有結論**：qwen2.5:3b TTFT 2218ms／總 3076ms 最快；gemma4 系列超出 4GB VRAM 遭 CPU 卸載慢 20~40 倍
- **延伸方法**：加入銀行工具後，量「/user_text 收到 → 工具執行完 → /llm_response 完整」的任務級延遲（比純生成延遲更貼近使用者體感）

## E5. ASR 雙版本比較 ⭐架構決策佐證

- **狀態**：未開始（Phase B，需實車麥克風陣列＋喇叭）
- **目的**：sherpa-onnx 自由語句 vs wheeltec 離線命令詞——用數據支撐「為什麼保留雙版本」
- **方法**：固定 10 句測試語（5 句命令詞語法內、5 句自由語句）× 安靜／背景人聲兩環境 × 每句 3 次；記錄 `/user_text` 輸出與說完到輸出的延遲
- **產出**：辨識成功率×延遲×詞彙彈性三維比較表；預期結論「命令詞快而窄、自由語句廣而慢」＋實測數字

## E6. 端到端延遲瀑布

- **狀態**：未開始（Phase C，需時間戳工具）
- **目的**：系統效能全景——「訪客走到機器人前到聽到迎賓語音要幾秒」
- **方法**：寫一個時間戳記錄節點（訂閱鏈上各話題記 `now()`）：影像幀 → `/face_recognition/result` → `/user_text` → `/llm_stream` 首 token → `/speech_text` → 音訊開始播放；跑 10 輪取分布
- **產出**：瀑布圖（各段耗時堆疊）＋瓶頸分析（預期 LLM 段最長，正好呼應筆電 GPU 分工）

## E7. Pi 資源占用 ⭐分工決策佐證

- **狀態**：未開始（Phase A 起每次實測順手收）
- **目的**：證明「Pi 輕量／筆電重運算」分工的必要性
- **方法**：`top -b -n 3` 或 `ps -C <node> -o %cpu,%mem` 記錄各節點；三組態：(a) 只跑輕量節點（底盤+HMI+橋接）(b) 加 vision (c) 全開；同時記 `vcgencmd measure_temp`（節流風險）
- **產出**：組態×節點的 CPU/RAM 表；預期「vision 上 Pi 後 CPU 逼近上限」→ 分工決策的數據來源

## E8. 導航精度

- **狀態**：未開始（Phase B，需實驗室場地）
- **目的**：硬體章節的量化數據；驗證阿克曼參數正確性
- **方法**：(a) 停止誤差：同一目標點導航 10 次，捲尺量停止位置與目標的偏差（縱向/橫向分開記）；(b) 迴轉半徑：最大轉向角低速畫圓，量直徑÷2 對照規格 0.80m（來源：docs/底盤與韌體分析.md）；(c) 選做：建圖尺寸誤差（地圖上量實驗室已知牆距 vs 捲尺實測）
- **產出**：誤差均值±標準差；迴轉半徑實測 vs 規格對照
