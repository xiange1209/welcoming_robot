## A. 這週最優先

### A1. 完成人臉辨識端對端驗證（ROS2 版）

目標：確認透過 `/face_registration/register` 服務註冊後，`face_recognition` 節點能正確辨識 VIP、黑名單、訪客，並發布到 `/face_recognition/result`。

執行順序：

- [ ] 在 RPi4 上 `colcon build --symlink-install` 並 `source install/setup.bash`
- [ ] 確認有相機節點在發布 `/image_raw`（若話題不同，用 `image_topic` 參數對齊）
- [ ] 啟動 `face_recognition` 節點（開啟除錯影像）
- [ ] 啟動 `face_registration` 節點
- [ ] 用服務呼叫註冊 1 位 VIP（本人站在鏡頭前，節點會採集 `num_samples` 張樣本）
- [ ] 用服務呼叫註冊 1 位黑名單
- [ ] 用 `ros2 topic echo /face_recognition/result` 驗證 VIP / 黑名單 / 陌生訪客三種判定
- [ ] 用 `rqt_image_view` 看 `/face_recognition/debug_image` 確認框色與標籤（VIP=綠框、黑名單=紅框、訪客=藍框）

建議命令：

```bash
# 建置與環境
cd ~/專題        # workspace 根目錄（內含 src/）
colcon build --symlink-install
source install/setup.bash

# 啟動人臉辨識節點（開啟除錯影像發布）
ros2 run smartnav_vision face_recognition --ros-args -p publish_debug_image:=true

# 另開終端機：啟動人臉註冊節點
ros2 run smartnav_vision face_registration

# 註冊 VIP（受測者需站在鏡頭前）
ros2 service call /face_registration/register smartnav_msgs/srv/RegisterFace \
  "{person_name: '測試VIP', gender: 'M', person_type: 'VIP', num_samples: 10}"

# 註冊黑名單
ros2 service call /face_registration/register smartnav_msgs/srv/RegisterFace \
  "{person_name: '測試黑名單', gender: 'M', person_type: 'BLACKLIST', num_samples: 10}"

# 若手動改過資料庫，可要求辨識節點重載快取
ros2 service call /face_recognition/refresh_cache std_srvs/srv/Empty

# 觀察辨識結果（person_uuid / person_name / gender / person_type / confidence / bbox）
ros2 topic echo /face_recognition/result

# 觀察除錯影像
ros2 run rqt_image_view rqt_image_view /face_recognition/debug_image
```

驗證證據至少要留：

- [ ] `/face_recognition/result` 的 echo 輸出（VIP、黑名單、訪客各一段）
- [ ] `/face_recognition/debug_image` 畫面截圖
- [ ] 誤判或漏判案例紀錄
- [ ] 當下使用的閾值（`recognition_threshold`，預設 0.6；`confidence_threshold`，預設 0.5）

### A2. 完成 RPi4 smartnav_llm → 筆電 Ollama 驗證

- [ ] 筆電本機用 `curl http://localhost:11434/api/tags` 確認 Ollama 正常、模型已 pull
- [ ] RPi4 上 `curl http://<筆電IP>:11434/api/tags` 確認跨機連線可通（注意 Windows 防火牆需放行 11434）
- [ ] RPi4 上啟動 `smartnav_llm`，`ollama_base_url` 指向筆電

建議命令：

```powershell
# 筆電（Windows PowerShell）
$env:OLLAMA_HOST = "0.0.0.0"
ollama serve          # 或重啟 Ollama 服務

```

```bash
# RPi4
curl http://<筆電IP>:11434/api/tags

# 啟動 LLM 節點（依 B10 實測結果，建議先用 qwen2.5:3b）

  ollama_base_url:=http://<筆電IP>:11434 \
  model_name:=qwen2.5:3b

# 另開終端機：發送測試文字

```

驗證證據至少要留：

- [ ] RPi4 上 `curl /api/tags` 成功回應

### B10. LLM 模型效能比較（✅ 已完成 2026-07-02）

已於 2026-07-02 在筆電上實測完成。環境：Windows 筆電 RTX 3050 Laptop 4GB VRAM、Ollama 0.24.0、`scripts/benchmark_llm_models.py`、每模型 1 次暖機 + 5 次正式、串流 `/api/chat`、銀行場景 system prompt、temperature 0.1、num_ctx 512。

| 模型（Ollama tag） | 基底 | 首次輸出平均 (ms) | 總生成平均 (ms) | 最快 (ms) | 最慢 (ms) | 成功率 |
|---|---|---|---|---|---|---|
| gemma3-bank | gemma3:4b | 2303 | 4700 | 4583 | 4755 | 100% |
| gemma4e2b-bank | gemma4:e2b | 42235 | 47909 | 44838 | 50206 | 100% |
| gemma4e4b-bank | gemma4:e4b | 95066 | 107123 | 100231 | 123839 | 100% |
| qwen2.5:3b | qwen2.5:3b | 2218 | 3076 | 3017 | 3196 | 100% |

關鍵解讀：gemma4 系列模型檔（7.2GB / 9.6GB）超出 4GB VRAM，Ollama 卸載到 CPU 導致慢 20~40 倍；gemma3:4b（4GB）與 qwen2.5:3b（1.9GB）能塞進 VRAM。在此硬體上，**qwen2.5:3b 最快，gemma3:4b 次之，gemma4 系列不適合即時互動**。
